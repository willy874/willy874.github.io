過往的軟體開發之中，不同的職務溝通都依靠某個特定的文件或規範的窗口。但上下文理解其實常常有許多問題，甚至跨工具、跨協定、跨團隊模式等等也充滿變數。近期在個人的 Side Project 我和團隊導入了一個獨特的模式，符合 AI 時代的做法。

團隊中會有不同工作部位，每個工作部位互相溝通都會利用不同工具，Jira, Figma, API Doc… 各種，但成堆成山要學習的工具、看不懂的各種文件、戰國時代般的溝通接口、難以遷移的技術債，這些都是困難。

但有沒有一套溝通的接口是歷經年代遷移都不容易變化？又沒有一套工具可以基於這套通用的協定去滿足溝通上下文的滿足？

有，這套通訊協定就是自然語言的文件。
而現在風行的 AI Agent 就可以基於這套些協定滿足各團隊的工具與技術差異，彌補實作上的誤差。看得懂的定義來自 AI 的驗證，當 AI 認為不再有模糊空間，那就是客觀的「清楚」，這就更容易被管理。

所以最近我開始用 Skill 作為需求傳遞的根基，從 Product => Design => Frontend => Backend => QA 整條溝通的 Bridge 由 AI 交管，文件等於通訊協定，還不用自己寫，減少思考與用詞的落差。

雖然不一定適合每個人，但我覺得這是很有趣的方法，分享出來。
