Code review 是一件很重要的流程，但很多人會對於這個流程有很多意見。它既是管理程式品質的重要流程，也是讓團隊相互熟悉風格的方法。同時被詬病的消耗很多時間，也會造成很多團隊紛爭與爭吵。

Code review 的目的是什麼？

我並不認為它能管理品質，也不是審核誰寫的好或不好，更不是拿來對風格指指點點。

Code review 我認為是這樣的：

- 檢查有沒有粗心的問題。比方說錯字、沒有清乾淨的 log 等等。（如果有資源在 CI 檢查更好）
- 互相了解彼此的撰寫邏輯，並詢問與說明設計理念。
- 多一層防呆機制，避免不小心直接點擊到合併。
- 確認是否有改動到不屬於的 feature 範圍，或是主管確認是否改動到專案主體架構功能。

但是我覺得並不是這樣：

- 檢查撰寫程式的風格閱讀性（這種東西要嘛用 ESLint，要嘛團隊要自己定義規範）
- 確認與討論解決方案（這種事要嘛在開發前就要討論，不然就等後續優化重構再說）

如果是有足夠資源的團隊，那或許可以 Code review 去管理品質。但多數團隊太過於小，需要逐步優化，花大量時間在 Code review 上無疑是浪費有限資源。

但如果因為認為 Code review 是大團隊才需要做的，我認為大錯特錯。只要是 2 人以上需要協作，那就需要 Code review 去維繫協作。
