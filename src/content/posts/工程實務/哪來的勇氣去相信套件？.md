很多工程師莫名無條件相信套件，但你知道套件不一定寫的好嗎？

過去我挑選套件我一定會去原始碼大致上讀過一次，用最快的速度判斷是否有使用價值。其實300行的程式有時候關鍵可能只是其中一個 function ，面對複雜的套件需求它得寫得很彈性，但面對你的業務需求可能根本不需要。其實重點在如何快速掌握一段程式碼的重點在哪，而不是「因為我不懂，用就對了」。如果你只是名 Junior ，或許我能理解你看不懂，但如果是工作2年以上的工程師，我會覺得你應該慢慢去培養自己讀原始碼的習慣。

我的資料夾中有一個地方都是去知名套件 clone 下來的套件庫，經常是我用了一個不熟悉的函式前我要先理解它的運作原理，我會先簡單了解一次，不要跟現有寫法和狀態引起衝突。一來可以提高對程式碼的閱讀經驗，二來減少未來改寫修復的時間。

為最不喜歡用的套件類型是以 Vue 或 React 為前綴的套件。其實你知道嗎？當初寫這種耦合性套件的人對你正在使用的框架不一定很了解，所以寫出來的東西也不見得漂亮。我可能沒有博愛到經常去 github 發 PR，所以對自己公司的程式品質管控可能就得更加嚴謹。

至於有哪些我只能說很多，也有那種很多人用的套件，但寫得真的很可怕⋯⋯我只能說，小心使用⋯⋯
