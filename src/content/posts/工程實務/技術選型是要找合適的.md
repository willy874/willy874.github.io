最近在社群看到一個議題，就是寫 Vue 的人現在各種推「Composition API 很棒，有新專案都應該使用」這樣的觀點，這我覺對保持否定態度。團隊的技術選型各種層面都得要考慮，不是一昧追求新技術，最潮的寫法。

前些陣子 YouTuber 「Alex 宅幹嘛」直播有提到，optional api 是有價值的，人不能忘本，其實我蠻認同的。現在寫 Vue 的工程師大概9成是寫 Optional API 出來的，它背後有很深遠的學習價值。

實質來說，Composition API 最大的優勢是可以把同一個元件功能關聯的東西組合進 composable function 之中，這樣閱讀可以依照資料流和邏輯去分離和排列。這樣的特性在越大的專案越是明顯，複雜的邏輯被抽象化為更微小的函式，因而更好被重複運用與拆分。

很顯然這件事並不是一件容易的事情，對於對程式還沒有這麼熟悉的人其實不容易做到。那自然就會誕生出很可怕的程式碼。因為不知道怎麼分類，所以邏輯會整坨攪豁成一團，閱讀不但跳來跳去，而且參數各種跨域溝通。

如果選擇 Optional API ，強制把變數分類擺放，大幅度控制了風格的可控性。當然你也可能會覺得 composition api 也不會多難閱讀，稍微學一下就好了吧？但問題在於給 junior 使用，會有超多誤操作，整個就是大大的偏離原本設計風格。

技術選型沒有什麼比較好，我只能說，依照環境特性，自認有「適合的」選擇。
