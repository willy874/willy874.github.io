多數談到架構多半只會想到後端的系統架構，但前端也是有所謂的架構，但要注意什麼東西？多半大家認知的前端架構不過是使用 Builder Tool 建構一個專案，一些資料夾結構和粗略的資料流。

但實際上前端的課題和後端、服務系統等等完全不同，自然要關注的焦點也截然不同。前端的介面是一個 context 很複雜的地方，會有各種狀態和事件的交雜，如何去管理這些就是前端架構要煩惱的。

前端架構要關注的重點：

- 易被進行測試，有明確的測試節點，也具備可被測試的互動性
- 不同狀態的資料，都可以在 UI 上被良好被顯示
- 多國語系在不同的語言上，顯示不會導致含義曲解與破版
- 相同業務邏輯下，利用參數與狀態來達到不同的樣式版面
- 使用者做了任何動作或發生任何事件都不阻礙畫面操作，正確引導使用者下一步行為
- 跨 feature 的溝通能很好解決耦合問題，模組之間的 API 能清楚表達
- 每一個功能命名的規範可以區分不同功能的性質，能在不了解細節前初判用途

以上也只是隨口提及，其實還有很多。議題上雖然都覆蓋在軟體工程領域，但不同於以往後端所在意的東西，前端關注點截然不同。
