近年越來越常聽到設計師主動請求則採用 Storybook，什麼時候開始在設計師的領域被傳開來了？

我經歷數間公司，大部分公司導入 storybook 的失敗都在於不重視元件庫迭代。輕易修改，輕易加功能，文件規格沒有工程師以外的人協助維護。AI 當道的現在，其實維護 Storybook 成本更降低了，那些 .stories 檔案其實也是給 AI 爬蟲很好的 Context，我認為現今更需要去維護這樣的元件庫工具。

必須說，storybook 並不適合所有的團隊，畢竟它維護成本相當高。依照經驗，storybook 的 Use case 和 Component api documentation 的維護是非常費心的，如果不是專案反覆需要重用元件，並不這麼適合。所以大致上可以歸類，接案公司其實不適合使用 storybook ，除非有長期維護的專案與產品，那在於人員交接與接手上就會帶來很大的價值。

我其實推崇，但也不盲推，真的看需要。如果公司產品非常專案性質，介面沒有固定的調性，那其實意義也不大，甚至連元件庫都不需要。
