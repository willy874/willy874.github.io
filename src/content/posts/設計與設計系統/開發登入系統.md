我想很多人開發過線上系統時肯定都開發過「登入系統」，除了帳號密碼登入、帳號註冊、二次驗證、忘記密碼、第三方登入各種方案的組合拳。

這是關係使用者對於系統的第一印象，關係甚遠。

使用者未來會開始使用？

使用者註冊後帳號是否安全？

未來系統漸進式拓展功能是否有空間？

有沒有辦法第一眼呈現出產品調性？

會不會跟其他類似系統搞混？

相信身為一個產品設計一定很在乎這些事。

然而，登入系統可大可小，每個系統在乎的細節有所不同，每個公司期待的業務邏輯也有所不同。

但是，我今天想講的是。

無論是當工程師的時候還是當產品設計師的時候都有遇到這件事。

工程是開發登入系統多少會想要用，不管是 Auth0, AWS Cognito, Firebase, Keycloak 等等，都是很好用的工具，省時省力。但問題在，要客製化時就會感覺到不方便。

比如説要細緻化 UI Flow 時，受到各種 API 的限制和架構的綁定，要改變它的流程時就會感受到不便利。

比如説要做自己品牌調性的系統 UI 時，還是得轉成純 API 溝通的架構。

身為工程師當然覺得難道不能照著 Library 的基礎走嗎？

身為產品設計當然覺得我都投資成本規劃了，難道不能完全做到一樣嗎？

不同位置有不同的視角，每個位置看到的資訊都是有道理的，最後對於整個產品帶來最高的貢獻決策很重要。

不管你是產品設計還是工程師，我真覺得站在不同的角色上才能看到更全面的資訊，挺身做出最有商業價值的決策。
